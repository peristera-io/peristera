# Disaster-recovery runbook

What to do when the Scaleway node — the single machine running everything
(ADR-0020) — is lost or destroyed. Written for the 3-a.m. operator: follow it
top to bottom. The companion rehearsal (below) is the proof this works; a
backup that has never been restored is a hypothesis (#77).

Scope: total node loss (instance/disk destroyed). A plain `scw instance server
stop`/start keeps the volume and needs none of this. For a single broken
tenant, jump to "Per-tenant restore" and skip the platform steps.

## What survives node loss (and where)

| What | Where | Custody |
| --- | --- | --- |
| Postgres (platform + every tenant): WAL + daily base | `s3://<backups>/zitadel-db`, `s3://<backups>/tenants/<slug>` | SCW key |
| Kamara blobs (encrypted chunks), nightly 03:30 | `s3://<backups>/tenants/<slug>/blobs` | SCW key |
| Tenant DEKs, age-encrypted, nightly | `s3://<backups>/tenants/<slug>/secrets.tar.gz.age` | **age private key** |
| Platform secrets (masterkey, admin-client, openfga) | Scaleway Secret Manager (primary, tofu-generated) + `s3://<backups>/platform/secrets.tar.gz.age` (second chain) | tofu state / age key |
| Images | ghcr.io | GitHub |
| Manifests, bootstrap, this runbook | git | GitHub |

You personally must hold, outside the cluster and outside the repo:

1. the **age private key** (password manager) — without it every escrowed DEK
   is noise and tenant files are gone even with perfect bucket copies;
2. the **SCW credentials** (`.env.scaleway`) and the tofu state (Object
   Storage backend for real deploys);
3. access to the **EuroDNS** account (peristera.lu records are manual).

## What does NOT survive (know before you need it)

- **Tenant CRs** (slug, domain, enabled apps) live only in etcd. Keep the
  list; they are recreated by hand in step 4. Four lines of YAML each.
- **Tenant status** — most is re-derived, but `status.Issuer` is load-bearing
  for the legacy custom-apex tenant (see step 4's warning).
- **App DSN / S2S secrets** — regenerated by the reconciler; S2S keys are
  re-minted against the restored Zitadel (AddAppKey), DSN passwords are reset
  by CNPG's role reconciliation to match the fresh app Secret.

## Full restore procedure

### 0. Stop and collect

age key, SCW creds, tofu state access, the tenant list. Do not `tofu destroy`
anything — the buckets and Secret Manager entries are the recovery source and
are independent of the node.

### 1. Rebuild the infrastructure

```sh
cd deploy/scaleway
set -a && . ../../.env.scaleway && set +a
tofu plan    # should show: recreate instance (+ IP attach), nothing destroyed
tofu apply
```

Do NOT let a drifted plan replace `scaleway_secret.platform` entries or the
buckets — if the plan wants to, stop and reconcile state first: replacing the
masterkey secret makes the restored Zitadel DB unreadable (the escrow copy in
the bucket is then the only way back).

DNS: `*.peristera.app` records are republished by external-dns once the
platform is up. peristera.lu A records (EuroDNS) must be repointed manually if
the Flexible IP changed.

### 2. Restore the platform database (before bootstrap's Zitadel step)

`bootstrap.sh` applies `manifests/cnpg-zitadel.yaml`, which creates a FRESH
`zitadel-db`. For DR you must recover it from the bucket instead:

1. Move the old backup prefix aside first (a recovered cluster must never
   archive WAL into the history it recovered from):

   ```sh
   rclone moveto :s3:$BACKUPS_BUCKET/zitadel-db :s3:$BACKUPS_BUCKET/pre-dr/zitadel-db
   ```

2. Edit the `Cluster` in `cnpg-zitadel.yaml` (one-off, don't commit) to add:

   ```yaml
   spec:
     bootstrap:
       recovery:
         source: pre-dr
     externalClusters:
       - name: pre-dr
         barmanObjectStore:
           destinationPath: s3://<backups-bucket>/pre-dr/zitadel-db
           serverName: zitadel-db
           endpointURL: https://s3.<region>.scw.cloud
           s3Credentials:
             accessKeyId: { name: scaleway-secret, key: SCW_ACCESS_KEY }
             secretAccessKey: { name: scaleway-secret, key: SCW_SECRET_KEY }
   ```

   (Keep the existing `backup:` block — the recovered cluster then archives
   fresh WAL to the original, now-empty path.)

3. Run `./bootstrap.sh` (exports incl. `BACKUP_AGE_RECIPIENT`). ESO re-syncs
   the platform secrets from Secret Manager — the SAME masterkey — so the
   recovered DB decrypts. If Secret Manager is also gone, restore the secrets
   from the escrow first:

   ```sh
   rclone copy :s3:$BACKUPS_BUCKET/platform/secrets.tar.gz.age /tmp/
   age -d -i <age-key-file> /tmp/secrets.tar.gz.age | tar xz -C /tmp/platform-secrets
   # recreate the Secret Manager entries (or the k8s Secrets directly) from these
   ```

4. `zitadel-db-dsn`: bootstrap rebuilds it from the fresh `zitadel-db-app`
   Secret if missing; CNPG resets the `zitadel` role password to match. Verify
   Zitadel comes up and `https://iam.<domain>` serves.

### 3. Per-tenant restore

For each tenant, BEFORE creating its Tenant CR (the reconciler is create-only:
what already exists is adopted, what is missing is created fresh — so
pre-create everything that must come from backup, or you get empty-fresh):

```sh
SLUG=acme; NS=tenant-$SLUG   # namespace naming: see controller (nsName)

# 0. move the old backup prefix aside (same WAL-history rule as the platform)
rclone moveto :s3:$BACKUPS_BUCKET/tenants/$SLUG :s3:$BACKUPS_BUCKET/pre-dr/tenants/$SLUG

# 1. namespace + backup creds
kubectl create ns $NS
kubectl create secret generic backup-s3 -n $NS \
  --from-literal=ACCESS_KEY_ID=$SCW_ACCESS_KEY --from-literal=ACCESS_SECRET_KEY=$SCW_SECRET_KEY

# 2. the DEK — from escrow, BEFORE the reconciler can generate a fresh one
rclone copy :s3:$BACKUPS_BUCKET/pre-dr/tenants/$SLUG/secrets.tar.gz.age /tmp/$SLUG/
age -d -i <age-key-file> /tmp/$SLUG/secrets.tar.gz.age | tar xz -C /tmp/$SLUG/
kubectl create secret generic kamara-dek -n $NS \
  --from-file=dek=/tmp/$SLUG/kamara-dek/dek

# 3. recovered CNPG cluster named "db" (reconciler adopts it)
#    — recovery from pre-dr/tenants/$SLUG serverName db, keep a backup: block
#    pointing at the original tenants/$SLUG path (fresh WAL history), same
#    shape as the platform recovery above. Wait for cluster Ready.

# 4. blob PVC + content — pre-create the PVC (name kamara-blob, RWO, 10Gi),
#    then fill it from the bucket with a one-off pod running the backup image
#    with the copy REVERSED:
#      rclone copy :s3:$BACKUPS_BUCKET/pre-dr/tenants/$SLUG/blobs /mnt/blob
```

Then create the Tenant CR (slug/domain/apps as before).

**Legacy custom-apex tenant (peristera.lu) warning:** its OIDC issuer is
`https://peristera.lu` — recorded only in `status.Issuer`. A recreated CR has
empty status, so the reconciler would look up the instance at
`lu.peristera.app`, find nothing, and CREATE A NEW EMPTY INSTANCE. Before the
controller reconciles the CR (scale the control-plane to 0 while doing this),
patch the status:

```sh
kubectl patch tenant lu --subresource=status --type=merge \
  -p '{"status":{"issuer":"https://peristera.lu"}}'
```

Tenants provisioned post-ADR-0021 need no patch: their issuer host equals
`<slug>.peristera.app` and the reconciler adopts the restored instance via
`InstanceIDByDomain`.

### 4. Verify — do not declare recovery until

- `blobverify` passes for every tenant (below) — every file decrypts;
- each tenant's issuer serves OIDC discovery (`/.well-known/openid-configuration`);
- a real login + file download works on one tenant;
- fresh WAL is archiving (new `wals/` objects under the ORIGINAL prefix) and
  the nightly jobs run green.

## Verifying a restore: blobverify

`kamara/cmd/blobverify` proves restitution end-to-end: it walks every object's
latest manifest in the restored DB, decrypts every chunk with the escrowed DEK
(AEAD tag + plaintext-hash recheck), and checks sizes. Run it from the repo
against a port-forwarded restored DB and a local copy of the restored blobs:

```sh
kubectl port-forward -n $NS svc/db-rw 5433:5432 &
rclone copy :s3:$BACKUPS_BUCKET/pre-dr/tenants/$SLUG/blobs /tmp/$SLUG/blobs
cd kamara && go run ./cmd/blobverify \
  -dsn "postgresql://<user>:<pw>@localhost:5433/kamara" \
  -blobs /tmp/$SLUG/blobs -dek /tmp/$SLUG/kamara-dek/dek \
  -tenant <issuer-host>   # e.g. acme.peristera.app; peristera.lu for legacy lu
```

Exit 0 and `N objects verified, 0 failed` is the proof.

## Rehearsal

Rehearse into a scratch namespace against the LIVE bucket (read-only for the
source prefix: the scratch cluster gets no `backup:` block and a different
name, so it never writes where it read from):

1. scratch ns + `backup-s3` secret;
2. CNPG `Cluster` `dr-rehearsal` with `bootstrap.recovery` from
   `tenants/<slug>` (serverName `db`), no backup block;
3. download blobs + escrow, `age -d` the DEK, run `blobverify`;
4. tear the scratch namespace down.

Rehearse after any change to backup/restore machinery, and at least monthly
until this is automated. Record each rehearsal in the worklog.

## Known gaps (tracked)

- Full-platform DR bring-up (steps 1–3 end-to-end) has not been rehearsed —
  only the per-tenant data path (DB + blobs + DEK + verify). #77 stays open
  until a full rehearsal on a scratch node happens.
- CNPG `Database` CR adoption of pre-existing databases in a recovered
  cluster is per CNPG's declarative semantics — verify on first rehearsal.
- Blob backup is nightly: RPO for file content is up to 24h (Postgres WAL is
  continuous). Acceptable interim; #21 (blobs in S3 natively) closes it.
- One SCW key guards backups + DNS + storage (#77): a scoped, bucket-only
  key for the backup jobs is a follow-up.
- Chunk object keys are BLAKE3(plaintext) — bucket readers can run
  confirmation-of-content probes (ADR-0001 trade-off, noted in the deploy
  README).
